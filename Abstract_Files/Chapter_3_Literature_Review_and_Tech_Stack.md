# Chapter 3: Literature Review and Technology Stack

## 3.1 Overview of the Literature
The development of modern web applications, particularly in the domain of food technology and e-commerce, is built upon decades of research in distributed systems, human-computer interaction, and database optimization. A thorough literature review was conducted to evaluate existing legacy systems against modern architectural paradigms. Historically, web applications relied heavily on Server-Side Rendering (SSR) tightly coupled with monolithic architectures (such as the LAMP stack: Linux, Apache, MySQL, PHP). While monolithic systems provided simplicity in deployment, they scaled poorly under high concurrent loads, primarily because scaling required vertical hardware upgrades rather than horizontal distribution.

The transition toward Single Page Applications (SPAs) and asynchronous, non-blocking APIs marked a critical pivot in web engineering. The literature heavily favors decoupled architectures where the frontend presentation layer operates independently of the backend data processing layer, communicating strictly via REST (Representational State Transfer) or GraphQL APIs. This project adopts this modern paradigm, leveraging the MERN stack (MongoDB, Express.js, React.js, Node.js) to achieve high availability, rapid development cycles, and superior client-side performance. 

This chapter explicitly details the technology stack chosen for the "CraveBites" project, deeply analyzing the software engineering concepts, performance metrics, and architectural justifications behind each chosen framework and library.

## 3.2 The Runtime Environment: Node.js
At the foundation of the backend architecture is Node.js, an open-source, cross-platform runtime environment built on Google Chrome's highly optimized V8 JavaScript engine. Node.js fundamentally shifts the paradigm of server-side programming by utilizing an event-driven, non-blocking I/O model. 

### 3.2.1 The Event Loop and libuv
Traditional web servers, such as Apache HTTP Server, operate on a multi-threaded model where each incoming client request spawns a new thread or process. This thread consumes dedicated RAM (often several megabytes) and CPU cycles. Under high concurrent loads, such as a localized spike in food orders during dinner hours, thread allocation becomes a severe bottleneck, leading to thread starvation and memory exhaustion (commonly known as the C10K problem).

Node.js elegantly resolves this concurrency issue through its single-threaded Event Loop, powered by the C-based `libuv` library. When a client requests data (e.g., fetching a restaurant menu), Node.js does not block the main execution thread while waiting for the database query to resolve. Instead, the asynchronous database operation is delegated to a worker pool (managed by libuv), freeing the main thread to immediately accept and process subsequent HTTP requests. Once the database returns the query results, a callback function is pushed onto the event queue, which the Event Loop eventually picks up and executes, returning the HTTP response to the client. This deeply asynchronous nature allows "CraveBites" to handle tens of thousands of concurrent connections on relatively modest cloud hardware.

## 3.3 The API Framework: Express.js (Version 5.2.1)
While Node.js provides the raw network primitives via its `http` module, routing complex applications using raw Node.js leads to verbose and hard-to-maintain code. Express.js is utilized as the minimal and flexible web application framework that provides a robust set of features to engineer web and mobile APIs. The project specifically utilizes the latest iteration of Express (v5.2.1), which introduces significant architectural improvements over its predecessor.

### 3.3.1 Middleware Architecture
Express operates on a sophisticated middleware architecture. A middleware function is a procedural hook that has access to the request object (`req`), the response object (`res`), and the `next` middleware function in the application’s request-response cycle. "CraveBites" implements an extensively layered middleware pipeline. 
1. **Application-Level Middleware:** Utilized for global parsing. The application uses `express.json()` to parse incoming JSON payloads, replacing legacy tools like `body-parser`.
2. **Security Middleware:** CORS (Cross-Origin Resource Sharing) heavily regulates which client domains are permitted to interact with the API, preventing unauthorized origin requests.
3. **Authentication Middleware:** Custom middleware aggressively checks for valid, cryptographically signed JSON Web Tokens (JWT) in the `Authorization` headers before permitting access to protected routes, such as the administrative dashboard or user checkout endpoints.

### 3.3.2 Native Asynchronous Error Handling
A historic limitation of Express v4 was its inability to automatically catch rejected Promises inside asynchronous route handlers. Developers were forced to wrap every `async/await` controller in a `try...catch` block or utilize external libraries like `express-async-handler`. The transition to Express v5 structurally integrates native Promise resolution. If an asynchronous route handler throws an error or returns a rejected promise, Express automatically intercepts it and forwards it to the centralized error-handling middleware, creating a significantly cleaner, more declarative, and robust codebase.

## 3.4 The Database Layer: MongoDB Framework
Relational databases (SQL) heavily rely on rigid, predefined tabular schemas. While excellent for normalized financial records, SQL struggles with the polymorphic, deeply nested, and rapidly evolving data requirements of a complex food ordering platform. "CraveBites" utilizes MongoDB, a distributed NoSQL document database.

### 3.4.1 BSON Document Structure
Unlike traditional rows and columns, MongoDB stores data in dynamic, multi-dimensional BSON (Binary JSON) documents. This provides absolute structural flexibility. For instance, a `MenuItem` document can contain deeply nested arrays of customization options (e.g., `["extra cheese", "no onions"]`) without requiring complex, performance-draining `JOIN` operations across multiple separate tables. The BSON serialisation format ensures that these complex documents are stored compactly and can be rapidly transmitted across the network.

### 3.4.2 Object Data Modeling (ODM) via Mongoose (Version 9.2.1)
While MongoDB's flexibility is powerful, applications still require data validation at the application layer to aggressively prevent corrupt data insertion. The project implements Mongoose (v9.2.1), an elegant MongoDB object-modeling tool designed to operate within an asynchronous environment. 

Mongoose provides an uncompromising, schema-based solution to model application data. It enforces strong type casting, highly specific validation logic, and complex query building. In "CraveBites," strict Mongoose schemas explicitly define the structure for standard entities. For example, the `User` schema mandates that the `email` field must be statistically unique across the entire collection and match a specific Regular Expression standard for email addresses. Furthermore, Mongoose middleware (pre and post hooks) is utilized to automatically hash user passwords via the bcrypt algorithm prior to saving the document into the database, guaranteeing that raw passwords never inadvertently persist in database logs.

## 3.5 The Frontend View Layer: React 19 Ecosystem
The frontend paradigm represents the most dynamic layer of the technology stack. Historically, interfaces were rendered as single HTML files updated via localized, imperative DOM manipulation (e.g., jQuery). For a complex, highly interactive food delivery platform requiring dynamic cart calculations and instant UI updates without complete page reloads, imperative DOM manipulation leads to "spaghetti code" and severe performance degradation.

### 3.5.1 The React Declarative Paradigm
React 19, adopted for the frontend infrastructure, implements a declarative programming model heavily centered around an abstraction known as the Virtual DOM. Instead of interacting with the notoriously slow browser DOM API directly, React constructs a lightweight representation in highly optimized memory. When application state mutates (e.g., a user clicks "Add to Cart"), React calculates the differences (the 'diff') between the new Virtual DOM and the previous version, and subsequently commits only the optimal, absolutely minimal batch of precise changes to the actual browser DOM. This reconciliation algorithm (Fiber) ensures absolute maximal rendering performance.

### 3.5.2 Advanced Hooks and Modern API Utilization
React 19 vastly expands upon the React Hooks API. "CraveBites" strictly adheres to functional component architecture, avoiding legacy class-based implementations entirely. Standard hooks such as `useState`, `useEffect`, and `useMemo` are extensively utilized to maintain complex localized state (such as tracking form input values) and to manage component side-effects (such as deeply interacting with the backend API via Axios). Furthermore, the application relies on deeply configured React Context APIs for global state dissemination, distributing deeply crucial session data like user authentication status globally down the component tree without suffering from detrimental prop drilling.

### 3.5.3 Single Page Application Routing
To navigate the platform seamlessly, `react-router-dom` (v7.13.0) is implemented. This powerful library completely intercepts browser navigation events, dynamically unmounting and rendering deeply nested component hierarchies without triggering a slow, disruptive server-side page reload. This client-side routing approach guarantees that the platform feels identical to a native operating system application, providing an instantaneous psychological response to user interactions.

## 3.6 Frontend Build Tooling: Vite (Version 8)
For modern React applications, the build logic and bundling strategy is paramount to developer velocity and production performance. Historically, tools like Webpack dominated this space; however, as projects scaled, Webpack's bundling time increased exponentially, leading to deeply frustrating developer experiences.

The project explicitly adopts Vite (v8) as its core build pipeline. Vite represents a monumental shift in web architecture. During the development environment phase, Vite fundamentally abandons the concept of bundling all modules together. Instead, it serves source code over native ES Modules directly to the browser. The incredibly fast `esbuild` dependency pre-bundles node modules extremely quickly written natively in Go. When a file is edited, Vite employs incredibly efficient Hot Module Replacement (HMR) specifically over isolated WebSocket connections, updating precisely the altered module natively within the live browser in mere milliseconds regardless of overall application size.

For production, Vite delegates the bundling process natively back to Rollup, producing highly optimized, heavily minified JavaScript chunks natively implementing automatic code-splitting strategies designed beautifully to significantly reduce the incredibly critical initial browser payload size, highly elevating search engine metrics and application interactivity metrics such as "Time to Interactive" (TTI).

## 3.7 Styling and Component Architecture: Tailwind CSS (Version 4.2.0)
Historically, cascading style sheets (CSS) were notoriously incredibly difficult to scale within expansive projects. Global CSS namespaces frequently led cleanly to unexpected specificity collisions, cascading overrides, and massive amounts of totally unused rules, dramatically bloating stylesheet file sizes.

To architect an ultra-premium visual aesthetic elegantly, "CraveBites" utilizes Tailwind CSS (v4.2.0). Tailwind is not an incredibly highly opinionated UI kit. Rather, it is a deeply mathematical, heavily constrained utility-first CSS framework natively generating distinct CSS classes corresponding directly precisely to single CSS properties (e.g., `flex`, `pt-4`, `text-center`, `rotate-90`).

### 3.7.1 The JIT (Just-In-Time) Compiler Paradigm

## 3.8 Security Tooling and Financial Infrastructure Integration

### 3.8.1 Cryptography: Bcrypt.js (v3.0.3)
In modern web applications handling sensitive user credentials, storing plaintext passwords is a critical security vulnerability and an industry-wide violation. The "CraveBites" architecture implements the `bcryptjs` (v3.0.3) library to rigorously hash incoming user passwords during the authentication flow. 

Beyond rudimentary cryptographic hashing (such as legacy SHA-256 integrations), `bcrypt` is explicitly designed to be computationally expensive. It utilizes the Blowfish cipher and implements a configuration known as the "work factor" or "cost". By iteratively processing the cryptographic algorithm (e.g., 10 or 12 salt rounds), `bcrypt` deliberately slows down the hashing execution time to approximately 100-300 milliseconds per operation. While this latency is imperceptible to a legitimate user attempting to log in, it becomes astronomically prohibitive for malicious actors attempting to execute offline brute-force or dictionary attacks against a compromised database dump. 

Furthermore, `bcrypt` automatically implements internal dynamic key salting. A unique cryptographic salt (a randomized high-entropy string) is generated and automatically prepended to each user's password fundamentally before the hashing sequence initializes. This architectural design directly neutralizes the threat of precomputed "rainbow tables," ensuring that even if two users share the exact same raw plaintext password, their resulting cryptographic hashes within the MongoDB database will be exceptionally distinct.

### 3.8.2 Access Control: JSON Web Tokens (JWT)
The implementation of JSON Web Tokens (JWT version 9.0.3) establishes an uncompromising, stateless authentication infrastructure within the application ecosystem. Contrary to traditional monolithic frameworks that heavily rely on stateful server-side sessions stored directly in aggressive system memory (e.g., Express Sessions) or distributed caching layers (e.g., Redis), the MERN stack thrives on decentralized, stateless microservice interactions.

During a successful login request, the Express.js server verifies the candidate password against the generated `bcrypt` hash. Upon successful algorithmic validation, a unique JWT is cryptographically created utilizing a highly secure, heavily guarded `HMAC SHA256` signing algorithm. This token comprises three specific mathematical segments: the Header (identifying the signing algorithm), the Payload (encapsulating critical, non-sensitive user metadata such as the `_id` and structural `role`), and the definitive Signature. 

Because the digital signature mathematically guarantees that the internal payload cannot be externally tampered with or modified without instantly invalidating the token, the backend API can confidently execute authorization decisions without requiring an expensive secondary horizontal database lookup. The React 19 frontend securely stores this JWT, attaching it to the `Authorization` header utilizing the `Bearer` schema in all subsequent Axios HTTP requests directed at protected `CraveBites` API endpoints. 

### 3.8.3 Payment Gateway Infrastructure: Razorpay Integration
Finally, the "CraveBites" ecosystem seamlessly integrates modern, enterprise-grade digital financial processing natively utilizing the Razorpay API network. The transaction lifecycle is meticulously partitioned into two distinct security phases: order instantiation and payment verification. 

When a consumer initiates the final checkout process, the Node.js backend calculates the ultimate aggregate total natively on the secure server side—bypassing any potentially manipulated frontend data metrics. The backend subsequently communicates directly with the Razorpay cloud infrastructure via a server-to-server HTTP request to systematically construct a definitive Razorpay Order ID. 

Once the user completes the client-side UI payment utilizing their preferred digital methodology (UPI, Credit/Debit, Net Banking), Razorpay transmits a highly specific payment payload natively back to the client. This payload is intimately forwarded back towards the Node.js API, where an `HMAC SHA256` cryptographic signature verification routine algorithmically confirms the transaction's absolute legitimacy utilizing the strict Razorpay Merchant Secret Key. Only upon this rigorous cryptographic clearance does the database successfully update the customer's chronological Order History as definitively 'Paid', ensuring that phantom transactions, synthetic fraud, and localized network interception methodologies are fundamentally impossible. 

## 3.9 Summary of Technology Decisions
In conclusion, the meticulous curation of the complete technological ecosystem heavily underpinning the "CraveBites" application natively solves the profound scaling and developmental limitations inherently found within legacy enterprise architectural models. By combining the exceptional frontend declarative rendering performance of React 19 and Vite, the absolute aesthetic consistency naturally provided by Tailwind CSS, the immensely scalable asynchronous microservice capabilities of Node.js and Express 5, and the incredibly profound structural flexibility of MongoDB, this project stands functionally as a highly modern, production-optimised representation of cutting-edge full-stack software development.
